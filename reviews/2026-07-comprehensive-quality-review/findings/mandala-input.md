# Mandala input/dispatch layer — deep code-quality review findings

Scope: src/application/app/ (all), src/application/platform/, src/application/keybinds/, src/application/macros/, src/main.rs, src/application/mod.rs. Conventions read: CODE_CONVENTIONS.md, TEST_CONVENTIONS.md, CONCEPTS.md §5-§6, lib/baumhard/CONVENTIONS.md.

## Architecture assessment

The dispatch architecture is in genuinely good shape at its core: the Action funnel (`dispatch_compatible` → `cross_dispatch::apply_*` → console-verb mutation cores) is real and holds for nearly every Compatible action; the `ThrottledInteraction` trait + adaptive `MutationFrequencyThrottle` is a well-conceived shell for continuous gestures; scene rebuilds use a principled signature-hashed §B2 in-place-mutator dispatch with a generic handle-role core; the macro privilege gate is single-sourced and heavily tested; and grapheme discipline via `grapheme_chad` is honored in every editor. The debt concentrates at two seams. First, the native/WASM boundary: WASM's mouse handlers re-implement double-click, create-orphan, wheel-zoom, and click-outside-commit bodies inline (the wheel path bypassing the keybind table entirely), so the "one vocabulary" promise of the funnel silently doesn't hold in the browser. Second, the modal-editor family: the edge-label and portal-text editors are ~230-line structural mirrors of each other, with three parallel modal-steal blocks and three parallel click-outside-commit blocks, and the color picker's commit/cancel Actions never reach the funnel at all — violating the codebase's own stated rule that commit/cancel are user-named effects. Around these sit an SSOT hazard in `KeybindConfig` (a field/resolve-table split that has already produced one shipped drift bug), a stale CONCEPTS §5 that misdescribes the frame heartbeat, and a persistent pattern of doc-comment merge damage (dangling fragments attached to the wrong functions).

### M2. WASM wheel zoom bypasses the keybind funnel and duplicates the zoom body
Severity: P1 | Category: ssot | Confidence: high
Files: src/application/app/run_wasm/event_mouse_wheel.rs:19-52; src/application/app/run_native.rs:418-489; src/application/app/dispatch/cross_dispatch/camera.rs:38-52
Evidence: WASM: `let factor = if scroll_y > 0.0 { 1.1 } else { 1.0 / 1.1 }; … renderer.process_decree(RenderDecree::CameraZoom { … }); rebuild_scene_only(…)` — no `action_for_gesture(WheelUp/WheelDown)` lookup, no `dispatch_compatible`. Native routes the same event through the funnel to `apply_zoom_step` (camera.rs:38-52), which owns the identical 1.1 factor. The `scroll_y` decomposition (`LineDelta`/`PixelDelta pos.y / 50.0`) is also copied verbatim (run_native.rs:422-425 vs event_mouse_wheel.rs:20-23).
Why it matters: §3 single dispatch funnel: "Every user-driven application-level behavior … dispatched via dispatch_action." `ZoomIn`/`ZoomOut` are classified `Compatible` and `dispatch_compatible` is already wired on WASM (keyboard + macros use it), so this is not blocked on convergence work — unlike the console/picker gaps it is NOT in WASM_CONVERGENCE.md's deferred list. Users who rebind or unbind WheelUp/WheelDown get native-only behavior; the 1.1 factor and the /50.0 divisor can drift independently. Also a behavior divergence: WASM does a synchronous `rebuild_scene_only` per wheel tick while native defers via the connection-geometry-dirty flag drain.
Fix: In `handle_mouse_wheel`, compute the gesture name from `scroll_y`, look up via `self.keybinds.action_for_gesture(…)`, and run `dispatch_compatible` through `input_context_core` (the ZoomIn/ZoomOut arms already exist). Extract `wheel_lines(delta: MouseScrollDelta) -> f64` as a shared plain-value helper used by both targets.
Effort: S

### M3. KeybindConfig: every action's binding is declared in 3+ hand-synced places; the drift has already shipped once
Severity: P1 | Category: ssot | Confidence: high
Files: src/application/keybinds/config.rs:41-288 (fields), 290-505 (Default), 517-921 (resolve() `sets` table + 22 push_parametric calls); src/application/keybinds/tests.rs:87-152
Evidence: Adding one bindable Action requires: a struct field (config.rs:41-288), a Default entry (290-505), an entry in the `sets` array or a `push_parametric` call (519-897), plus tests. Nothing ties them together: a field missing from `resolve()` compiles and the user's binding is silently dead. Fix: Collapse declaration to one table: a macro (`declare_keybinds! { undo: Undo = ["Ctrl+Z", "Undo"], … }`) that expands to the struct fields, Default impl, and `resolve()` arms; or, minimally, add a completeness test that constructs a config with a sentinel binding in every field via serde introspection (serialize `Default`, set every array to `["F24"]`, deserialize, resolve, assert bind count == field count).

### M4. ThrottledInteraction covers only the drain half — accumulate and commit-on-release are open-coded per variant
Severity: P2 | Category: api-design | Confidence: high
Files: src/application/app/throttled_interaction/mod.rs:126-193; src/application/app/event_cursor_moved.rs:144-189; src/application/app/event_mouse_click.rs:598-846,1049-1140
Evidence: The trait is `has_pending / throttle / drain / reset` only. The input side is 7 hand-written arms in `event_cursor_moved.rs`: five identical delta-accumulate bodies (`i.total_delta += delta; i.pending_delta += delta;` at 144-170) plus two identical cursor-overwrite bodies (171-189). The release side is 7 arms in `event_mouse_click.rs`'s left-release match (598-846), each with the same shape — flush residual pending, commit to model, push/skip undo, `scene_cache.clear()`, rebuild — of which only NodeResize/SectionResize were extracted (`finalize_node_resize_release` / `finalize_section_resize_release`, 1049-1140, whose own doc cites §5 for exactly this reason); MovingNode (598-656), MovingSection (657-696), EdgeHandle (703-739), PortalLabel (740-781), EdgeLabel (782-811) remain inline. The 6-line `has_pending`/`throttle` accessor boilerplate is also repeated 8× (7 drags + picker hover).
Why it matters: §5 duplication; §6 strategy pattern — the shape is plural and half-abstracted. CONCEPTS §5 sells the trait as "new throttled drags attach as one struct + one trait impl + one enum variant without growing the dispatch," but today a new variant must also grow two match ladders in two event files; the WASM drag port (planned per WASM_CONVERGENCE.md) would have to duplicate the release ladder again.
Fix: Add two trait methods with the two existing idioms as provided impl helpers: `fn accumulate(&mut self, input: DragInput)` where `DragInput = Delta(Vec2) | Cursor(Vec2)` (or split methods with default no-ops), and `fn commit_on_release(&mut self, ctx: DrainContext<'_>)`. `event_cursor_moved`'s Throttled arm becomes one line via `as_dyn_mut()`; the left-release match collapses to `Throttled(mut k) => k.as_dyn_mut().commit_on_release(…)`. The already-extracted finalize helpers become the NodeResize/SectionResize impls.
Effort: M

### M6. Color-picker commit/cancel/nudge Actions never reach the dispatch funnel — inconsistent with the codebase's own TextEdit/LabelEdit rule
Severity: P2 | Category: convention | Confidence: high
Files: src/application/app/color_picker_flow/key.rs:44-106; src/application/app/dispatch/native.rs:636-652,766-769; src/application/keybinds/action/mod.rs:421-445
Evidence: native.rs:637-641 states the project rule: "Commit/cancel are user-named effects (Esc / Enter / click-outside), NOT the §3 carve-out for literal Key character insertion — so they belong in the funnel," and TextEditCommit/Cancel + LabelEditCommit/Cancel are dispatched accordingly (event_keyboard.rs:94-97,133-136,174-177). But `PickerCancel`, `PickerCommit`, and the six `PickerNudge*` Actions are resolved and executed entirely inside `handle_color_picker_key` (key.rs:67-99); `dispatch_action` has no arms for them, so they fall to `_ => Unhandled` (native.rs:766-769). Consequence beyond consistency: a macro step `MacroStep::Action { PickerCommit }` is a silent no-op (macro loop counts it un-ran), while `TextEditCommit` from a macro works.
Why it matters: §3 funnel ("no second copy of Action body logic in a handler"); the codebase's own documented posture is applied to two of the three modals. Macros/plugins are the named trajectory and cannot drive the picker.
Fix: Move the eight Picker* arm bodies into `dispatch_action` (native match), and have `handle_color_picker_key` pre-filter exactly like the label/text editors: resolve action in `InputContext::ColorPicker`, dispatch any `Picker*` match through the funnel, keep only Copy/Paste/Cut modal-specifics + fallthrough local.
Effort: M
