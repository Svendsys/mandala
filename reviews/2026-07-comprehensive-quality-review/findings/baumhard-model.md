# Findings — baumhard mindmap model / loader / custom_mutation review

## Architecture assessment

The baumhard mindmap layer is a well-shaped, unusually well-documented data-model core: plain serde structs with no derived state, a deliberately thin loader, cascade helpers that (for zoom windows and font sizes) are genuine single sources of truth, and a custom-mutation carrier with a thought-through undo-snapshot contract and a serious backward-compat deserializer. Test density on serde round-trips is exemplary. The cracks are all of one family: knowledge that exists in more places than one. The palette subsystem is documented as the render-time color cascade but is in fact unwired (resolve_theme_colors has no production caller); the format docs — which claim to be authoritative — contradict the code (and each other) on text-run coverage, TextRun optionality, control-point coordinates, target-scope count, and canvas fields; section/size validation rules and messages are hand-mirrored byte-for-byte across maptool and the app crate instead of living in the model both depend on; the edge-color cascade is open-coded at six-plus sites while its zoom sibling got a proper helper; and the same interval arithmetic lives twice (TextRun ops vs ColorFontRegions). Performance posture is mostly conscious, but the model's O(n)-scan `children_of` quietly makes tree builds O(n²) and drag frames O(n·desc), and the loader's "cheap" legacy screen double-parses every styled map. Nothing here is rotten; almost everything below is the §5/§B0 discipline of collapsing two hand-maintained copies into one.

## Findings

### F1. Palette color cascade is documented as live but is unwired — `resolve_theme_colors` is dead in the render path
Severity: P1 | Category: ssot | Confidence: high
Files: lib/baumhard/src/mindmap/model/mod.rs:183-192; format/palettes.md:52-64; CONCEPTS.md:1088-1092; lib/baumhard/src/mindmap/scene_builder/node_pass.rs:152,245-270; lib/baumhard/src/mindmap/tree_builder/node.rs:80-114,173-188; lib/baumhard/src/mindmap/loader.rs:405-413 (only caller = test)
Evidence: palettes.md: "How a node resolves its colors: `resolve_theme_colors(node)` … the renderer falls back to the node's plain `style` colors" and "Editing a palette is a single-point change; every node using it updates on the next render." Repo-wide grep for `resolve_theme_colors` / `color_schema`: no scene_builder / tree_builder / renderer caller — node fill, frame, and text colors come exclusively from `node.style.*` + `resolve_var` (node_pass.rs:152; tree_builder/node.rs:80,106,175). `map.palettes` is consulted at render time only for border glyph cycling (`resolve_palette_cycle`, border.rs:1393). The miMind converter baked resolved colors into `style.*`, which is why themed maps still look right.
Why it matters: format/ claims authority; its central palette promise (single-point retheme) is false today. Editing `map.palettes` changes nothing on screen. `resolve_theme_colors` is a pub API whose only caller is its own test — a §B9 doc that lies about the system, and either dead code or an unfinished feature (§5: no half-features).
Fix: Either (a) wire `resolve_theme_colors` into node color resolution in tree_builder/node.rs + scene_builder/node_pass.rs (style fields = no-schema fallback, exactly as palettes.md describes), or (b) rewrite palettes.md/CONCEPTS to say `color_schema` is carried-but-unconsulted and mark the helper as the seam. (a) is the honest close per §5.
Effort: M (a) / S (b)

### F2. Loader accepts `parent_id` cycles; `is_hidden_by_fold` / `all_descendants` / `is_ancestor_or_self` then hang or stack-overflow in interactive paths
Severity: P1 | Category: correctness | Confidence: high
Files: lib/baumhard/src/mindmap/loader.rs:43-74 (no structural validation); lib/baumhard/src/mindmap/model/mod.rs:134-148,151-162,167-179; crates/maptool/src/verify/tree.rs:26-42 (cycle check exists only in the optional CLI)
Evidence: `is_hidden_by_fold` walks `parent_id` with no visited set (`while let Some(pid) = current_id { … }`) — a 2-cycle `a→b→a` with neither folded never terminates. `collect_descendants` recurses through `children_of`; the same cycle is infinite mutual recursion → stack overflow (abort, uncatchable). validation.md:23 admits "A cycle makes `all_descendants` loop forever" but only `maptool verify` checks. The console `open` verb loads arbitrary files at runtime; scene build calls `is_hidden_by_fold` per node (node_pass.rs:130).
Why it matters: §9 — interactive paths must not panic; a hang is worse. macros.md's threat model: "opening any `.mindmap.json` from an untrusted source IS a privilege event" — a hostile or corrupt map is a one-line DoS.
Fix: Reject cycles in `load_from_str` (one O(n) visited-set pass over `parent_id`, error naming the offending node — same posture as the zero-sections rejection), or give the three walkers a depth cap of `nodes.len()`. Loader-side rejection is cheaper and keeps walkers lean.
Effort: S

### F3. `ControlPoint` rustdoc claims absolute canvas coordinates; the implementation (and every other doc) uses offsets from node centers
Severity: P1 | Category: docs | Confidence: high
Files: lib/baumhard/src/mindmap/model/edge.rs:568-572; lib/baumhard/src/mindmap/connection/mod.rs:119-134; CONCEPTS.md:1229-1246; format/schema.md:248
Evidence: edge.rs: "Stored in canvas-space coordinates (absolute, not relative to the endpoints)." connection/mod.rs:120: `let qp = from_center + Vec2::new(control_points[0].x as f32, …)` ("Control point is offset from source node center"); :133-134 cubic: `from_center + cp0`, `to_center + cp1`. CONCEPTS §3 and schema.md ("Bezier offsets") agree with the code. The same doc block also claims the struct is "`Copy`-free only because `f64` fields keep parity with the JSON precision" — `f64` is `Copy`; the sentence is nonsense.
Why it matters: §B9 — "A doc comment that lies about its function is worse than no doc comment at all." This is the pub model type a plugin author reads via cargo doc; absolute coordinates produce curves that track node movement wrongly.
Fix: Rewrite the doc to offset-from-endpoint-center semantics (cp[0] relative to from-center, cp[1] to to-center; one point = quadratic promoted to cubic), and derive `Copy` or drop the bogus sentence.
Effort: S

### F6. Edge/label/portal color cascade open-coded at six-plus sites across two crates; the zoom and font-size cascades got SSOT helpers, color did not

### F7. The same half-open-interval arithmetic is implemented twice: `text_run_ops` (Vec<TextRun>) and `ColorFontRegions` (BTreeSet<ColorFontRegion>)
Severity: P1 | Category: duplication | Confidence: high
Files: lib/baumhard/src/mindmap/model/text_run_ops.rs:56-70 (split_at), 209-240 (splice_range: cut+shift+fill), 124-142 (merge); lib/baumhard/src/core/primitives.rs:202-220 (split_and_separate), 239-249 (shift_regions_after), 279-302 (insert_regions_at), 329-362 (shrink_regions_after), 489-544 (`Range` with overlaps/magnitude/push_right)
Evidence: `splice_range`'s delete path (split at both ends → retain outside → shift later runs by `replacement_len - removed`) computes exactly `shrink_regions_after`'s case table (fully-left / fully-right / fully-inside / spans-cut / left-partial / right-partial), and its insert path is `shift_regions_after` + gap-fill. Both are grapheme-cluster half-open `[start, end)` span tables with the no-overlap sorted invariant; text_run_ops re-implements the math on bare `usize` pairs instead of the existing `core::primitives::Range`. The two also drift at edges: `split_and_separate` can leave a degenerate `[k, k)` region when the insertion point equals a region start (primitives.rs:207-211: `region.range.end = range.start` with `start == range.start`), while text_run_ops never produces empty runs (`debug_assert!(r.start < r.end)`).
Why it matters: §5 / §B0 — same interval algebra, two vocabularies, hand-kept in sync; the model↔tree bridge (`region_to_text_run` in document/custom/sync.rs) already converts between them, so a semantic drift (like the empty-region edge case) becomes a round-trip bug. Unification is natural: both live in baumhard.
Fix: Concretely — (1) make `TextRun` use `core::primitives::Range` (or expose `range()` accessors); (2) extract the classification/shift math into payload-generic helpers on `Range` (e.g. `Range::classify_against_cut(cut) -> SpanCase`, `Range::shifted(delta)`), or a small generic span-table module parameterized over `T: HasRange + Clone`; (3) reimplement `shrink_regions_after`/`insert_regions_at`/`shift_regions_after` and `splice_range`/`split_at` on that core. Keep the two container types (Vec for serde-ordered runs, BTreeSet for the hot region path) — unify the arithmetic, not the storage. ColorFontRegions is hot (§B7 benches exist); reuse must be `#[inline]`-friendly and allocation-free there.
Effort: L

### F8. `children_of` is an O(n) full-map scan+sort; recursive consumers make tree builds O(n²) and drag frames O(n·desc)
Severity: P2 | Category: performance | Confidence: high
Files: lib/baumhard/src/mindmap/model/mod.rs:122-130 (children_of), 151-162 (all_descendants = recursive children_of); lib/baumhard/src/mindmap/tree_builder/node.rs:311 (`build_children_recursive` — one children_of call per node ⇒ O(n²) per tree build); src/application/app/throttled_interaction/moving_node.rs:117 (all_descendants per throttled drag flush per dragged node); src/application/document/custom/mod.rs:565-595 (per mutation apply); src/application/document/mutations/tree_cascade.rs:49; crates/maptool/src/export.rs:29-61 (consumer already built a private one-pass `ChildIndex` to dodge this, with a comment naming `children_of` as the trap)
Evidence: `children_of` filters `self.nodes.values()` (full HashMap walk) and sorts per call. `build_children_recursive` calls it for every visited node — the testament map is 252 nodes ⇒ ~252 full scans per `build_tree()`; `build_tree` runs at init, on every structural rebuild (src/application/app/scene_rebuild.rs:375, drain_frame.rs:40), and inside a cursor-move path (event_cursor_moved.rs:779). `moving_node.rs:117` runs `all_descendants` (O(n·desc)) on each throttled drag flush when dragging a subtree. export.rs:29: "`MindMap::children_of` is the obvious call here but…" — proof a consumer already needed and hand-rolled the index.
Why it matters: §B1 — budget for a phone; the cost is quadratic in map size, felt on every structural rebuild and during subtree drags. For a 5k-node map, one tree build is ~25M HashMap-iteration steps. The model API shape (per-call scan) invites this at every new call site; maptool's private index is a §1 "second implementation" symptom.
Fix: Add a one-pass child index to the model API (e.g. `MindMap::child_index() -> ChildIndex` building `HashMap<Option<&str>, Vec<&MindNode>>` sorted once, with `children_of`/`descendants_of` reading it), use it in tree_builder's recursion, custom-mutation collection, drag descendants, and replace maptool's private copy. Keep the convenience `children_of` for one-shot callers with a doc note on cost (it already lacks a §B9 cost note — the struct doc promises "see each method for its per-call cost" but `children_of`/`root_nodes`/`all_descendants` docs omit the O(n)/O(n²) figures).
Effort: M

### F9. `is_hidden_by_fold` (O(depth) hash-walk) is recomputed per element per scene/tree build across 8+ sites; CONCEPTS claims it "runs once per scene build"
Severity: P2 | Category: performance | Confidence: high
Files: lib/baumhard/src/mindmap/scene_builder/node_pass.rs:130 (per node); scene_builder/connection.rs:146 (per edge ×2 endpoints); scene_builder/label.rs:127,216; scene_builder/portal.rs:412; scene_builder/section_frame.rs:68; scene_builder/builder.rs:358,386; tree_builder/mod.rs:138, tree_builder/node.rs:313, tree_builder/border.rs:74, tree_builder/portal.rs:179; src/application/document/hit_test.rs:213,518,633; CONCEPTS.md:1345-1347
Evidence: Each call re-walks the full ancestor chain via `nodes.get`. One `build_scene_with_cache` (per frame during drags — moving_node.rs:123, edge_label.rs:84, edge_handle.rs:104) pays N·depth for node_pass plus 2·E·depth for connections plus the label/portal passes — the same ancestor chains re-walked four-plus times per frame. Inside `build_children_recursive` the check is redundant by construction (ancestors above the recursion point are already known-visible; only the direct parent's `folded` bit can differ).
Why it matters: §B1 mobile budget: redundant hash-chain walks per frame on every visible element. CONCEPTS' cost claim ("O(depth) per node and runs once per scene build") undersells reality by ~4×.
Fix: Compute a per-build hidden-set once (single O(n) pass: a node is hidden iff parent hidden or parent folded — one HashMap<&str, bool> memo) and thread it through the scene/tree builder passes; in `build_children_recursive` replace the call with a `parent_folded` bool passed down. Update the CONCEPTS cost sentence.
Effort: M

### F10. Loader "cheap substring screen" triggers a full second `serde_json::Value` parse on every styled map; the cost doc is wrong
Severity: P2 | Category: performance | Confidence: high
Files: lib/baumhard/src/mindmap/loader.rs:32-42 (cost claim), 53-59, 83-85 (`has_legacy_marker` checks `"text_runs":`); maps/testament.mindmap.json (254 `"text_runs":` keys — every load double-parses)
Evidence: Post-section files legitimately contain `"text_runs":` inside every styled section, so `has_legacy_marker(json)` is true for essentially every non-trivial map, and `detect_legacy_shape` re-parses the entire file into `serde_json::Value` on every load. The doc comment claims "Cost: one typed parse on the happy path. The legacy-shape detection (Value walk) runs only when the typed parse fails OR … OR when the cheap substring screen flags a dropped field" — the screen flags ~always.
Why it matters: §B9 lying cost doc; load path is "felt every map load" (loader.rs:19) and doubles peak memory (full Value tree) for large maps on mobile.
Fix: Drop `"text_runs":` from the marker (node-level `text_runs` without `text` is vanishingly rare, and the zero-sections condition already catches real legacy nodes since legacy nodes lack `sections`), or scope the marker check to node objects via a streaming check; at minimum fix the cost doc.
Effort: S

### F11. `flat_mutations` treats a RepeatWhile with a matches-nothing predicate as always-true; guard clause is dead code via `||`/`&&` precedence
Severity: P2 | Category: correctness | Confidence: high (behavior) / med (real-world reach)
Files: lib/baumhard/src/mindmap/custom_mutation/mod.rs:297-311; lib/baumhard/src/gfx_structs/predicate.rs:150-186 (`Predicate::new()` matches nothing; `test` returns false for empty fields)
Evidence: the guard is `p.always_match || p.fields.iter().all(|(_, c)| matches!(c, Comparator::Equals(_))) && p.fields.is_empty()` — parsed as `a || (b && c)`; `b` is vacuously true exactly when `c` (is_empty) is true, so the whole guard reduces to `always_match || fields.is_empty()`, and the `.all(Equals)` clause can never influence the result (dead code, evidence of a mangled intent). Consequence: a hand-authored `{"Instruction":{"instruction":{"RepeatWhile":{"fields":[],"always_match":false}}, …}}` — which `Predicate::test` says matches zero elements, and which mod.rs:99-102's own doc calls "matches nothing — a footgun" — is accepted by the flat-apply path and its inner Macro's mutations are applied to the whole scope set. The comment above the arm says "Only honour `always_true` predicates … falling through to `None` makes that case warn at the apply site" — the code does the opposite for the empty-fields case.
Why it matters: Semantic inversion between the (current) flat-apply path and the (named-trajectory) walker path for the same JSON; undo is still safe (snapshot covers the scope) but the visible behavior differs from the AST's meaning and from the file the author wrote.
Fix: Tighten the guard to `p.always_match` only (matching the comment), so match-nothing predicates fall to `None` → the existing "non-flat mutator" warn fires; delete the dead `.all()` clause; add a test for the empty-predicate shape.
Effort: S

### F12. `TargetScope::covers_reach` assumes anchor-rooted application, but Parent/Siblings (and per-target anchoring generally) anchor the mutator at each target — Children/Siblings arms approve pairings whose snapshot is narrower than the touch set
Severity: P2 | Category: api-design | Confidence: med
Files: lib/baumhard/src/mindmap/custom_mutation/mod.rs:384-401 (covers_reach), 320-382 (MutatorReach/mutator_reach); lib/baumhard/src/mindmap/custom_mutation/scope.rs:18-23 ("`Parent` and `Siblings` … the application layer is responsible for iterating the right set of targets and anchoring the mutator at each of them"); src/application/document/custom/mod.rs:396-520 (flat-apply anchors at every affected node), 555-597 (snapshot set)
Evidence: `TargetScope::Children | TargetScope::Siblings => reach <= MutatorReach::Children`. Under per-target anchoring (the documented model for Parent/Siblings, and what flat-apply does for every scope), a `MapChildren`-reach mutator anchored at each sibling touches each sibling's children — nodes absent from the Siblings snapshot (`collect_affected_node_ids` returns siblings only). Today this is latent: `flat_mutations` returns `None` for `MapChildren` roots so the walker shapes never execute, but `covers_reach` is precisely the safety net for the future walker path (mod.rs:159-176 wires it as the apply-time warning), and its Children/Siblings arms encode the wrong anchoring model. Also: `Siblings` on a root node silently resolves to the empty set (custom/mod.rs:582-595 — `parent_id: None` ⇒ no siblings, though other roots are arguably siblings); no doc states this.
Why it matters: CONCEPTS §4 names the undo-snapshot equivalence "the load-bearing detail: if a mutation's target_scope is too narrow, undo will not fully reverse it." The gate that guards that detail is wrong for two arms the moment the walker path (explicitly a named next step, custom/mod.rs:279-281) lands.
Fix: Decide the anchoring contract per scope and encode it in `covers_reach`: if per-target anchoring stays, `Children`/`Siblings` must require `reach == SelfOnly`; if anchor-rooted walking is the future contract for Children, document that Siblings/Parent remain per-target and split the arms. Document the root-node Siblings behavior either way.
Effort: S (code) — the decision is the work

### F15. TextRun serde: docs present `bold/italic/underline/font/size_pt/color` as optional formatting metadata, but all are hard-required — text-runs.md's own example fails to deserialize
